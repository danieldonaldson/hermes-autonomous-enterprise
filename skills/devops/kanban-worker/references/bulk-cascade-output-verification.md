# Bulk Cascade Recovery — Worked Example (May 2026)

> Reference file for `kanban-worker` skill.
> A real-world example of a 38-task bulk cascade caused by MiniMax HTTP 429 rate limiting,
> where some tasks had completed their work before the protocol violation.

## The Cascade

**Date:** 2026-05-14
**Event:** Bulk provisioning of 38+ kanban tasks across 8+ profiles
**Common error on all tasks:**
```
worker exited cleanly (rc=0) without calling kanban_complete or kanban_block — protocol violation
```

## Root Cause

All workers were spawned with `provider=deepseek model=deepseek-v4-flash` configured, but the
routing layer was mapping them through `provider=minimax base_url=https://api.minimax.io/anthropic`.
MiniMax's Token Plan (designed for interactive use) returned HTTP 429 under the concurrent
worker load generated by the dispatcher.

Agent.log showed:
```
ERROR API call failed after 3 retries. HTTP 429: The Token Plan is designed for individual,
interactive developer workflows. Traffic is currently high—please retry shortly.
```

Followed by credential pool exhaustion:
```
credential pool: no available entries (all exhausted or empty)
```

## Output Verification Check

The key insight: some workers ran for minutes (successfully completing their work) before
hitting the rate limit on a final summarization or cleanup turn.

**How it was done:**

1. Sampled 5 tasks across different profiles — the designer and finance tasks were picked
   because they had the longest runtimes (482s and 181s respectively).

2. Read each task body with `kanban_show` to find expected output paths:
   - `t_cc678a58` (Designer): body said "Save to: `docs/design/listing-upload-flow.md`"
   - `t_4159d423` (Finance): body said "Save at: ~/Work/yethu/docs/finance/break-even-analysis.md"

3. Verified with `search_files`:
   - `/home/daniel/Work/yethu/docs/design/listing-upload-flow.md` — EXISTS
   - `/home/daniel/Work/yethu/docs/finance/break-even-analysis.md` — EXISTS

4. Checked remaining 16+ marketing/research/CPO/engineering tasks — zero output files found

**Result:**
- **2 tasks** had completed work → candidates for `kanban complete`
- **36 tasks** had no output → need provider fix + re-dispatch

## Commands Used

```bash
# Step 1: List all blocked tasks
hermes kanban list --status blocked

# Step 2: Check individual task details
hermes kanban show t_cc678a58

# Step 3: Verify output existence
search_files(pattern='listing-upload-flow*', path='/home/daniel/Work/yethu/docs')
search_files(pattern='break-even-analysis*', path='/home/daniel/Work')

# Step 4: Check running duration from event log
hermes kanban show t_cc678a58 | grep -E 'spawned|protocol_violation'
# → spawned at 15:46, protocol_violation at 15:54 = 8 minutes (strong B2 candidate)

# Step 5: Check batch of potential outputs across all marketing tasks
# (looked for: growth-metrics.md, seo-strategy.md, support-metrics.md, content-calendar.md,
#  blog-drafts.md, community-targets.md, seed-messages.md, research/interview-guide.md,
#  cs-metrics.md, nps-log.md, link-building.md, design/landing-page.md,
#  design/search-discovery-flow.md, product/category-expansion-roadmap.md,
#  product/retention-conversion-mechanics.md, strategy/competitive-landscape.md,
#  product/mvp-search-spec.md)
# → None found across the entire batch
```

## Key Learnings

1. **Runtime is a proxy for B2 vs B1.** Tasks running >60s are candidates for output
   verification. Tasks exiting in <30s are almost certainly B1 (zero work done).

2. **Task body is the best output-path map.** Bodies written per task body templates
   almost always specify their output path explicitly (e.g. "Save to: docs/X.md").
   Parse the first 20 lines of the task body with `kanban_show` to find it.

3. **`search_files` works across permanent paths.** Scratch workspaces are deleted when the
   task is archived, but tasks that save to permanent docs/ directories leave a trace.
   Always check the destination path, not the workspace.

4. **Bulk-complete after verification, don't re-dispatch.** Completing verified tasks saves
   ~30 min of re-computation per task. Only re-dispatch tasks that have no output.

5. **The dispatcher doesn't distinguish B1 from B2.** Both produce identical `protocol_violation`
   events and identical `gave_up` outcomes. The output-verification step is the only way to tell
   them apart from outside the worker's runtime.
