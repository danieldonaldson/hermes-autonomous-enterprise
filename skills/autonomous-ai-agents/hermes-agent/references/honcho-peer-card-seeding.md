# Honcho Peer Card Seeding — Seeding the AI Peer with Structured Facts

## Overview

Honcho has two memory systems that serve different purposes:

| Component | What it stores | When it appears | How to populate |
|-----------|---------------|-----------------|-----------------|
| **Peer Card** | Static facts, behavioral rules, preferences | Immediately via API, injected every turn per `injectionFrequency` | `PUT /v3/workspaces/{wid}/peers/{pid}/card` |
| **Dialectic Conclusions** | Learned user model from conversations | After deriver processes conversation turns | Built automatically — needs ~1+ conversation cycles |
| **Representation** | Curated subset of conclusions for context injection | After dialectic runs | Generated by deriver from conclusions |

**Key insight:** `hermes honcho status` reports on dialectic conclusions ("No peer data yet"), NOT the peer card. A fully populated peer card and an empty dialectic store are both normal — they're different systems.

## Behavioral Rule: Prefer Honcho Over Built-in Memory

**When Honcho is configured as the memory provider (`memory.provider: honcho`), always use the Honcho peer card API for storing persistent facts — NOT the built-in `memory` tool.** The built-in tool has a ~2,200 char ceiling and stores facts in MEMORY.md. Honcho's peer card has no practical size limit (Postgres-backed) and is injected every turn per `injectionFrequency`. The built-in tool should only be used for tiny ephemeral rules when Honcho is unavailable.

## When to Use Direct Peer Card Seeding

The `hermes honcho identity <file>` command seeds the AI peer's identity, but it feeds into the **deriver's dialectic pipeline** — the data is queued for processing and doesn't become visible until the deriver runs. For **immediate** population of the peer card with structured facts, use the REST API directly:

- Migrating built-in `memory` tool entries to Honcho
- Setting persistent behavioral rules that should be available on first turn
- Seeding the AI peer with project/org-specific constraints

## Direct API Seeding

### Set the AI Peer Card

```bash
curl -s -X PUT "http://localhost:8000/v3/workspaces/default/peers/hermes/card" \
  -H 'Content-Type: application/json' \
  -d '{
    "peer_card": [
      "Rule one: concise, no fluff",
      "Rule two: always verify before telling user something is done",
      "Active cronjobs: CoS sync (4h), CEO strat (09:25), Review Router v2 (5min)"
    ]
  }'
```

The peer card is an **array of strings** — each entry is a standalone fact. Keep entries short and independently meaningful (they're injected as-is).

### Read Back the Peer Card

```bash
curl -s "http://localhost:8000/v3/workspaces/default/peers/hermes/card"
```

Returns:
```json
{
  "peer_card": ["Rule one: ...", "Rule two: ...", "..."]
}
```

### Verify via Hermes CLI

```bash
hermes honcho identity --show
```

This shows the AI peer's card entries. The user peer card is separate:
```bash
curl -s "http://localhost:8000/v3/workspaces/default/peers/daniel/card"
```

### Set Peer Card for a Different Target

The `target` query param lets the AI peer set a card *about* another peer (e.g., the AI's observations about the user):

```bash
curl -s -X PUT "http://localhost:8000/v3/workspaces/default/peers/hermes/card?target=daniel" \
  -H 'Content-Type: application/json' \
  -d '{"peer_card": ["User prefers concise responses", "Uses DeepSeek v4 Flash"]}'
```

## `hermes honcho identity <file>` — What It Actually Does

```bash
cat > /tmp/soul.md << 'EOF'
# AI Identity
- Tech Lead: reviews code ONLY, never writes code
- Up-or-Out: dual-dimension (KPI quality + attainment)
- Finance: no founder draw
EOF

hermes honcho identity /tmp/soul.md
# Output: Seeded AI peer identity from soul.md into session 'daniel'
#         Honcho will incorporate this into hermes's representation over time.
```

This command **does NOT set the peer card**. It stores the content as session-level data for the **deriver** to process. The deriver (running as a background worker) will:
1. Read the seed data
2. Generate dialectic conclusions from it
3. Incorporate those conclusions into the representation

**This takes time and can stall** — the deriver won't process representation
work units until accumulated tokens reach `REPRESENTATION_BATCH_MAX_TOKENS`
(default 1024). Small seed messages sit in the queue indefinitely. See
`references/honcho-deriver-troubleshooting.md` for how to diagnose stuck
deriver jobs and enable flush mode for immediate processing.

## Distinction: `hermes honcho identity --show` vs `hermes honcho status`

| Command | Shows | Empty means |
|---------|-------|-------------|
| `identity --show` | Peer card entries | No card set yet |
| `status` | Dialectic/conclusion status | Deriver hasn't processed any conversations |
| `/v3/.../peers/{id}/card` (GET) | Raw peer card from API | `null` or empty array |

Example of a healthy Honcho setup with populated peer card but no dialectic data yet:

```
$ hermes honcho status
  Connection...
  No peer data yet (accumulates after first conversation)
  OK

$ hermes honcho identity --show
  AI peer (hermes)
  Tech Lead: reviews code ONLY, never writes code.
  Up-or-Out: dual-dimension...
  ...

$ curl -s http://localhost:8000/v3/workspaces/default/peers/hermes/card
  {"peer_card": ["Tech Lead: ...", "Up-or-Out: ...", ...]}
```

All three outputs are consistent — the peer card is populated, but the dialectic engine hasn't run yet.

## Migration Path: Built-in Memory → Honcho Peer Card

When converting from the built-in `memory` tool (which stores facts in MEMORY.md with a ~2,200 char limit) to Honcho:

1. Read the current memory entries (from memory tool responses or `hermes memory status`)
2. Compose them into concise standalone strings
3. PUT them to the Honcho peer card API
4. Verify with `hermes honcho identity --show`
5. Optionally clear the built-in memory store

The peer card has **no practical size limit** (array of strings backed by Honcho's Postgres DB), unlike the built-in store's 2,200 char ceiling.

## Pitfalls

- **`hermes honcho identity` does NOT set the peer card** — it seeds the deriver pipeline. If the deriver isn't running, the seed sits unprocessed.
- **Peer card != dialectic conclusions** — seeing "No peer data yet" in status is normal even with a fully loaded peer card.
- **Peer card entries are strings** — not key-value pairs, not structured JSON. Each string is a complete fact.
- **`injectionFrequency: every-turn` in honcho.json** controls whether the peer card is injected into the system prompt. If set to a different cadence, the card data may not appear every turn.
- **The built-in `memory` tool and Honcho peer card coexist** — they're independent storage systems. Clearing one doesn't affect the other.
- **When Honcho is configured, prefer the peer card API over the built-in `memory` tool** for storing persistent facts. The built-in tool has a ~2,200 char ceiling and is for fallback only.

## Retrieval Paths: What Works With Zero-Vector Embeddings

When the embedding API is unavailable (e.g., DeepSeek has no embedding API — see `honcho-deriver-troubleshooting.md`), observations are stored with zero-vector embeddings. Different Honcho tools use different retrieval paths, with different success rates:

| Tool / API | Works with zero-vectors? | Notes |
|------------|--------------------------|-------|
| `honcho_search` | **Yes** | Uses `peer.context()` with search_query — non-semantic path works |
| `honcho_context` | **Yes** | Uses session context + recent messages — returns representation |
| `GET /v3/.../conclusions/list` | **Yes** | Returns all stored observations |
| `POST /v3/.../peers/hermes/representation` | **Yes** | Returns representation without semantic search |
| `honcho_reasoning` (dialectic) | **No** | Requires semantic search via embeddings — won't find zero-vector observations |
| `honcho_profile` | **No** | Uses dialectic to build peer card — needs semantic search |
| Peer card (`GET /v3/.../card`) | **Null** | Never populated automatically — must be set via PUT |
| `hermes honcho status` | **Partially** | Shows "No peer data yet" if card is null AND representation is empty |

**Workaround for `honcho_reasoning`:** If observations exist but semantic search can't find them, use `honcho_search` for direct retrieval instead. The dialectic's `minimal` reasoning level uses `search_memory` alone (semantic), so it won't find anything. Higher reasoning levels have more tool options but also break with the thinking mode issue (see `honcho-deriver-troubleshooting.md`).

## Setting Up Gemini Embeddings (Free Tier)

If you chose to use Gemini as the embedding provider (recommended over zero-vector fallback):

1. Get a free API key from https://aistudio.google.com/apikey
2. Update `~/Work/honcho/.env`:
   ```bash
   # Replace existing embedding settings
   EMBEDDING_MODEL_CONFIG__TRANSPORT=gemini
   EMBEDDING_MODEL_CONFIG__MODEL=gemini-embedding-001
   LLM__GEMINI_API_KEY=your-key-here
   ```
3. Restart all Honcho containers:
   ```bash
   docker compose -f ~/Work/honcho/docker-compose.yml restart
   ```
4. Verify:
   ```bash
   # Send a test message to trigger deriver
   curl -s -X POST "http://localhost:8000/v3/workspaces/default/sessions/daniel/messages" \
     -H 'Content-Type: application/json' \
     -d '{"messages": [{"content": "Test with Gemini embeddings.", "peer_id": "daniel"}]}'
   
   # Wait 30 seconds, then check
   curl -s -X POST "http://localhost:8000/v3/workspaces/default/conclusions/list" \
     -H 'Content-Type: application/json' -d '{}'
   ```
5. Once observations accumulate, run `hermes honcho status` — it should no longer show "No peer data yet"

The Gemini free tier includes `gemini-embedding-001` with generous rate limits (1,500 requests per minute). This is sufficient for personal Honcho usage.

Beyond the peer card, these endpoints are useful for managing Honcho state:

### Dialectic Chat — Query the Peer Card (No Deriver Needed)

The dialectic endpoint reads the peer card directly and responds via LLM, without waiting for deriver conclusions:

```bash
curl -s -X POST "http://localhost:8000/v3/workspaces/default/peers/hermes/chat" \
  -H 'Content-Type: application/json' \
  -d '{
    "session_id": "daniel",
    "query": "Summarize what you know from the peer card.",
    "reasoning_level": "low"
  }'
```

Parameters: `session_id` (scope), `target` (optional peer), `query` (required), `stream`, `reasoning_level` (minimal/low/medium/high/max).

### Seed Conversation Messages for the Deriver

The deriver processes conversation messages to build conclusions. Send messages to a session to seed processing:

```bash
curl -s -X POST "http://localhost:8000/v3/workspaces/default/sessions/daniel/messages" \
  -H 'Content-Type: application/json' \
  -d '{
    "messages": [
      {"content": "AI peer rules seeded.", "peer_id": "hermes"},
      {"content": "Acknowledged.", "peer_id": "daniel"}
    ]
  }'
```

### Check Queue Status

```bash
curl -s "http://localhost:8000/v3/workspaces/default/queue/status"
```

Shows pending/completed/in-progress work units per session. Useful for diagnosing stuck deriver jobs.

### Conclusions CRUD

```bash
# List all conclusions
curl -s "http://localhost:8000/v3/workspaces/default/conclusions/list" \
  -H 'Content-Type: application/json' -d '{}'

# Get AI peer representation
curl -s -X POST "http://localhost:8000/v3/workspaces/default/peers/hermes/representation" \
  -H 'Content-Type: application/json' \
  -d '{"session_id": "daniel"}'

# Get peer context (representation + card)
curl -s "http://localhost:8000/v3/workspaces/default/peers/hermes/context"
```

### Deriver Needs an API Key

The deriver (`docker compose ps` → honcho-deriver-1) processes the queue in the background. If `LLM_OPENAI_API_KEY` is missing from `.env`, the queue fills up with pending work units that never complete. Check:
```bash
curl -s "http://localhost:8000/v3/workspaces/default/queue/status"  # pending != 0 means stuck
docker logs honcho-deriver-1 --tail 20  # check for API errors
```
